## 二、🔴 致命问题（必须修正）

### 2.1 投标人身份造假风险

```
投标书框架.md L5: "投标人：中科应急科技股份有限公司（公章）"
```

- 你公司的真实主体是「袁马军·一人公司」，不是中科应急
- 招标方做背调时，企查查/天眼查一查就穿帮
- **必须决策**：是走"挂靠中科应急"还是"以自己的壳投标"
- 选挂靠 → 需要中科应急的授权书 + 公章
- 选自己 → 投标书全文替换公司名，去掉 98 人团队改为实际配置

### 2.2 98人/3,455人日团队不可兑现

```
需求梳理.md L12: "配置 98 名中级工程师，总工作量 3,455 人日"
```

- 这是大厂/系统集成商的投标话术
- 你一人公司，即使外包也凑不出 98 人
- **建议**：改为"核心团队 12-15 人 + 合作外包团队"，人日压缩到 1,200-1,500
- 否则一旦中标，甲方查团队履历时无法自圆其说

## 三、🟡 重要问题（建议修正）

### 3.1 前端渲染方案不一致

| 模块 | 技术方案.md | 需求梳理.md |
|------|------------|-------------|
| 舆情大屏 | 未明确 | ECharts + DataV |
| 人口热力 | Deck.gl (WebGL) | Leaflet + Canvas |
| 数据可视化 | 未展开 | ECharts |

- 同一个系统跑三套渲染引擎 = 维护灾难
- **建议**：统一为「ECharts(图表) + Deck.gl(地图/热力)」双引擎

### 3.2 BERT vs LLM 论述过时

```
技术方案.md L122: "GPT/LLM...单条推理延迟 500ms~2s（A100），无法满足实时吞吐"
```

- 这是 2024 年的数据。2026年 Qwen3-0.6B、Gemma 等小模型边缘推理延迟可到 50ms
- 投标书整体定位在"2025年的技术栈"，缺乏前瞻性
- 建议：BERT 主线不变，但增加「VLM/LLM 作为辅助研判通道」的演进路线

### 3.3 信创全栈验证证据缺失

```
技术方案.md L239: "已完成国产化软硬件全栈适配验证"
```

- 列举了鲲鹏+麒麟+达梦+东方通全套，但没有附任何测试报告/截图
- 评标专家大概率会问："在哪测的？有没有第三方报告？"
- 建议：找合作方补一份兼容性测试报告，哪怕只有麒麟+达梦

### 3.4 三大运营商数据对接前提不成立

```
需求梳理.md L139: "对接中国移动、中国联通、中国电信三大运营商的脱敏手机信令数据"
```

- 这不是技术能力问题，是**行政准入问题**
- 运营商信令数据需要通过省大数据局/政数局审批
- 建议增加前置条件："甲方需协调省级数据管理部门出具数据共享授权函"
