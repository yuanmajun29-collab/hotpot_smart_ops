## 二、开发阶段

### Phase 1: MVP 加强 (当前 — 2周)

| 任务 | 技术栈 | 状态 |
|------|--------|------|
| Count Anything部署到Jetson | Docker + Triton | 🔧 模型下载中 |
| 废料自动计数API | Python :8097 | 📋 待开发 |
| Hub集成Count结果 | FastAPI :8098 | 📋 |
| Dashboard计数卡片 | Vue | 📋 |

### Phase 2: 产品化 (2-6周)

| 任务 | 说明 |
|------|------|
| 多门店管理Hub | 门店ID隔离、数据分库 |
| 自动日报/周报 | AI Agent生成运营报告 |
| 告警系统 | 废料超标/汤位过低/SOP违规 → 微信推送 |

### Phase 3: 规模化 (6-12周)

| 任务 | 说明 |
|------|------|
| 模型自动更新管道 | GitHub Release → 门店自动拉取 |
| CCL跨场景增强 | 训练CBGB+CCLoss应对新环境 |
| 管理后台(SaaS) | 老板自助开/关门店、查看报表 |
| 付费/订阅系统 | 微信支付接入 |

## 三、技术选型

| 需求 | 选型 | 理由 |
|------|------|------|
| 边缘检测 | YOLO26-L | 成熟、TensorRT可加速 |
| 场景理解 | VLM (llama.cpp) | 已部署已验证 |
| 精确计数 | Count Anything | 文本引导，无需重新训练 |
| 跨场景泛化 | CCL (论文) | 训练增强，推理零延迟 |
| 后端API | FastAPI | 轻量、异步、已熟练 |
| 前端 | Vue 3 + ECharts | Dashboard已有 |
| 数据库 | SQLite (单店) / PostgreSQL (多店) | 渐进式 |
| 消息推送 | 微信企业号 + 公众号模板消息 | 直达老板 |

## 四、性能指标

| 环节 | 延迟 | 目标 |
|------|------|------|
| YOLO推理 | 30ms | <50ms |
