# 火瞳 — 产品解决方案

> 6大模块 × 全场景方案 | 基于创始背景+市场调研+现有代码

---

## 一、后厨损耗（覆盖85%·P0）

### 解决什么
火锅店后厨每天废弃的食材占原料成本15-25%，老板不知道、管不了。

### 怎么做
```
摄像头 → YOLO检测(8ms) → CLIP分类(50ms) → VLM语义(6.8s) → Count计数 → Hub → Dashboard
```

### 现有代码
- `edge/kitchen/inference/pipeline.py` — Stage注册表管线 ✅
- `edge/kitchen/inference/rules.py` — 阈值/降级矩阵 ✅

### 待补
- Count Anything推理集成（Docker容器） 📋
- 时序趋势存储+预警 📋

---

## 二、SOP规范（覆盖40%·P1）

### 解决什么
加盟店出品标准不一致，总代管不了每家店的操作规范。

### 怎么做
```
7工位状态机（汤底/切配/摆盘/蘸料/洗消/传菜/冷库）
   → Edge实时合规检测 → 违规自动告警
```

### 现有代码
- Hub端SOP引擎+清单 ✅
- Dashboard sop.html ✅

### 待补
- Edge端实时合规检测管线 📋（3-4天）
- 7工位状态机激活 📋

---

## 三、食材监管（覆盖25%·P1）

### 解决什么
进货缺斤少两、以劣充好、好食材无故废弃——每年漏掉的成本比浪费还多。

### 怎么做
```
电子秤(MQTT) → 重量数据
   +                  → 比对判断 → Hub → Dashboard → 告警
摄像头 → CV品质检测(YOLO)
```

### 现有代码
- Hub端 receiving API ✅（只存数据，不分析）

### 待补
- `edge/receiving/` 全新模块 📋（4-5天）
- Dashboard成本+进货看板 📋

---

## 四、前厅桌态（覆盖80%·P0）

### 解决什么
翻台率靠经验，空桌/需清理桌/在用桌状态靠人盯。

### 怎么做
```
摄像头 → plan_b(40ms 规则推断) / plan_a(190ms CLIP语义) → 桌态事件 → Hub → Dashboard
```

### 现有代码
- `edge/front_hall/inference/` — plan_a+plan_b双模式 ✅
- `edge/front_hall/inference/rules.py` — 桌态推理规则 ✅

### 结论
✅ 基本完整，试点可直接使用。

---

## 五、员工行为（覆盖5%·P2）

### 解决什么
员工交头接耳、着装不规范、行为懈怠——影响服务质量和品牌形象。

### 怎么做
```
摄像头 → YOLO检测人+PPE分类(着装) → 行为规则(交头接耳/停留时间) → Hub → Dashboard
```

### 现有代码
❌ 零代码，完全空白

### 待补
- `edge/staff_behavior/` 全新模块 📋（5-7天）
- Dashboard员工行为看板 📋

---

## 六、管理总览（覆盖75%·P0）

### 解决什么
总代/老板不在店里，想知道每家店实时状态，横向对比找问题。

### 怎么做
```
Hub聚合所有门店数据 → Dashboard cockpit → 总代Dashboard → 日报自动推送
```

### 现有代码
- Hub cockpit+regional API ✅
- Dashboard cockpit.html+regional.html+alerts ✅
- 日报自动生成 ✅

### 待补
- 多店对比增强 📋
