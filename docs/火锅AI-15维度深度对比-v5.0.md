## 02 技术栈

| 层级 | 海底捞 | 星巴克 | 火瞳 |
|------|--------|--------|------|
| **云平台** | 阿里云 DataWorks + MaxCompute | Azure + Databricks + Azure ML | 自建 Hub (FastAPI + PostgreSQL) |
| **数据库** | 阿里云 RDS / PolarDB | Azure SQL / Cosmos DB | PostgreSQL + SQLite (边缘) |
| **大数据** | DataWorks 数据治理 | Azure Data Lake + Delta Lake | 自建分析层 (948行) |

| **消息队列** | 阿里云 RocketMQ | Azure Event Hubs | Hub 事件总线 |
| **监控** | 阿里云 ARMS | Azure Monitor | device_health.py |

**关键差异**：两家巨头用顶级云厂商全家桶。火瞳的技术栈刻意轻量——FastAPI 不是 Databricks，但够用、能跑、好维护。万元盒子不需要百万云账单。

## 04 边缘计算策略

| | 海底捞 | 星巴克 | 火瞳 |
|------|--------|--------|------|
| **边缘推理** | 🟡 极少量试点 | ❌ 无 | 🟢 **核心架构** |
| **推理设备** | 未知 | 无 | Jetson Orin/Nano |
| **离线能力** | 弱 | 弱（需联网） | 🟢 **离线自运行 >24h** |
| **推理延迟** | — | — | plan_b <40ms |
| **模型更新** | — | — | OTA 增量下发 |
| **为何不做边缘** | 阿里云够快+不差钱 | 全球网络好+咖啡简单 | **火锅蒸汽+油污=必须边缘** |

**关键差异**：边缘计算不是"技术选择"，是**场景决定**。

## 06 数据架构

| | 海底捞 | 星巴克 | 火瞳 |
|------|--------|--------|------|
| **数据流向** | 门店→阿里云→DataWorks→报表 | 门店→Azure→Databricks→Dashboard | 门店→Jetson推理→事件→Hub→Dashboard |
| **数据存储** | 集中式云数仓 | Azure Data Lake | Hub PostgreSQL + 边缘SQLite |
| **实时性** | 准实时（分钟级） | 准实时 | 🟢 **<5秒告警** |
| **离线缓冲** | 无 | 无 | 🟢 store_forward 离线队列 |
| **视频数据** | 不上传（推测） | 不涉及 | 🟢 **只上传事件，视频留本地** |
| **分析层** | 阿里DataWorks | Databricks + Power BI | 🟢 自建948行分析引擎 |
| **数据湖** | MaxCompute | Delta Lake | — |

**关键差异**：两巨头数据架构是航母级（DataWorks/Databricks），火瞳是护卫舰级（自建FastAPI分析层）。护卫舰不跟航母比吨位，比的是：**实时性**（<5秒）和**数据隐私**（视频不上云）。
