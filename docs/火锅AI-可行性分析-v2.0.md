## 一、技术可行性：通用YOLO vs 火锅专用差距

### 1.1 现状技术栈（已实现）

| 组件 | 技术 | 成熟度 | 部署位置 | 已验证 |
|------|------|:---:|------|:---:|
| 目标检测 | YOLO26-L | ⭐⭐⭐⭐ | Jetson Orin / Mac | ✅ E2E 数据流 |
| 场景理解 | VLM（Jetson :8099） | ⭐⭐⭐ | Jetson Orin | ✅ |
| 前厅桌态 | YOLO + CLIP (plan_a/b) | ⭐⭐⭐⭐ | Jetson Orin | ✅ 40-190ms |
| 数据Hub | FastAPI :8098 | ⭐⭐⭐⭐⭐ | Mac/云端 | ✅ |
| Dashboard | Vue :3000 | ⭐⭐⭐⭐ | Mac | ✅ |
| 设备管理 | 注册/心跳/配置下发 | ⭐⭐⭐⭐ | 云端 | ✅ |

### 1.2 火锅专有场景的技术差距

通用YOLO模型（COCO预训练）能检测人/桌子/盘子，但火锅场景需要以下专用能力：

| 火锅专有场景 | 通用YOLO能否胜任 | 差距 | 需要的工作 | 难度 | 预估时间 |
|------|:---:|------|------|:---:|:---:|
| **蒸汽环境检测** | ❌ 失效 | 蒸汽遮挡→目标丢失/误检 | 红外+去雾算法，或蒸汽数据集微调 | 🟡 中 | 3-4周 |
| **食材切配规格识别** | ❌ 不适用 | 需判断"肉片厚度""菜量标准" | 定制分类模型 + 标注数据集(5000+样本) | 🔴 高 | 6-8周 |
| **食材以劣充好检测** | ❌ 不适用 | 需对比"应有品相 vs 实际品相" | CLIP语义对比 + 分级标注 | 🔴 高 | 8-12周 |
| **蘸料台监控** | ⚠️ 部分可用 | 可检测人/容器，无法判断蘸料种类/余量 | 小分类模型 + 容器液位检测 | 🟡 中 | 2-4周 |
| **锅底汤位检测** | ⚠️ 部分可用 | 可检测锅，无法判断汤量 | 专用分割模型或规则方法 | 🟢 低 | 1-2周 |
| **加汤时机判断** | ⚠️ 部分可用 | 汤位+服务员轨迹联合判断 | 多帧时序模型或规则引擎 | 🟡 中 | 2-3周 |
| **后厨7工位SOP** | ❌ 不适用 | 需状态机+多摄像头联动 | state machine + 时序推理 | 🟡 中 | 4-6周 |
| **进货区计量** | ❌ 不适用 | 需结合重量传感器或视觉估重 | CV估重算法 + 传感器融合 | 🔴 高 | 6-10周 |

### 1.3 差距总结与分级策略

```
已就绪（可直接商用）:
  ✅ 前厅桌态 (empty/dining/needs_cleaning)
  ✅ 基础后厨监控 (人员检测/活动检测)
  ✅ Dashboard + 告警

需1-2月补充（MVP增强）:
  🟡 蒸汽适配 (去雾/红外)
  🟡 7工位SOP状态机
  🟡 加汤时机检测
  🟡 蘸料台基础监控
  🟡 锅底汤位检测

需3-6月补充（专业版功能）:
  🔴 食材切配规格识别
  🔴 食材以劣充好检测
  🔴 进货区视觉计量
```

| 判断 | 依据 | 置信度 |
|------|------|:---:|

## 二、竞争壁垒分析

### 2.1 三层壁垒模型

基于竞品v2.0扫描，火瞳可构建三层壁垒：

```
                      ┌─────────────────────────────┐
                      │   第三层: 生态壁垒（远期）    │
                      │  火锅品牌深度集成 + 供应链   │
                      │  数据 → 成为行业基础设施      │
                      ├─────────────────────────────┤
                      │   第二层: 场景壁垒（中期）    │
                      │  火锅专有数据集 + 模型微调   │
                      │  蒸汽/切配/SOP/翻台闭环      │
                      ├─────────────────────────────┤
                      │   第一层: 时间壁垒（近期）    │
                      │  冯校长体系独占 + 先发案例   │
                      │  竞品窗口期 12-18个月        │
                      └─────────────────────────────┘
```

### 2.2 壁垒一：数据飞轮（越用越准）

| 飞轮环节 | 机制 | 护城河强度 |
|---------|------|:---:|
| 部署门店数↑ | 每家店每天产生5000-10000帧标注数据 | 🟢 每多一家店=数据资产+1 |
| 数据多样性↑ | 不同厨房布局/光线/蒸汽条件 | 🟢 泛化能力提升 |
| 场景模型精度↑ | 火锅专有数据集越大，专用模型越准 | 🟢 后发者无法复制 |
| 客户留存↑ | 模型越准→ROI越高→客户不愿换 | 🟢 切换成本高 |
| **飞轮启动条件** | ≥10家店、≥3个月数据、≥3种厨房布局 | **当前状态: 未启动(1家)** |

**飞轮启动前的时间窗口**: 竞品（尤其海康）可能在12-18个月内注意到火锅垂直机会。火瞳需在此期间完成10+店部署，启动数据飞轮。

### 2.3 壁垒二：火锅场景专用模型

| 壁垒维度 | 火瞳 | 海康/大华 | 商汤/旷视 | 创业公司 |
|------|:---:|:---:|:---:|:---:|
| 火锅蒸汽数据集 | ✅ 计划 | ❌ 无 | ❌ 无 | ❌ 无 |
| 7工位SOP状态机 | 📋 设计中 | ❌ 不适用 | ❌ 不适用 | ❌ 不适用 |
| 翻台率闭环分析 | ✅ 已实现 | ❌ 无此概念 | ❌ 无此概念 | ❌ 无此概念 |
| 食材损耗计量 | 📋 开发中 | ❌ 不做 | ❌ 不做 | ❌ 不做 |
| 后厨运营Knowhow | 🟢 有(冯校长) | 🔴 零 | 🔴 零 | 🔴 零 |
| 火锅品牌关系 | 🟢 冯校长总代 | 🔴 无 | 🔴 无 | 🔴 无 |
| **护城河深度** | ⭐⭐⭐⭐⭐ | ⭐ | ⭐ | ⭐ |

> 火锅Knowhow 不可被技术替代：海康可以做AI，但他们不知道"冯校长的切肉标准是3mm"。

### 2.4 壁垒三：集成深度

| 集成深度 | 当前状态 | 目标状态 | 护城河 |
|---------|:---:|------|:---:|
| 冯校长体系绑定 | 总代身份(独家优势) | 冯校长全国50-80店标配 | 🟢 竞品无解 |
| 浙江火锅连锁 | 待拓展 | 台州→全省500-800店 | 🟡 需6-12个月 |
| 运营流程耦合 | 独立系统 | 与收银/采购/排班系统联动 | 🟡 API集成=高切换成本 |
| 行业标准制定 | — | 火锅后厨AI行业标准(3年目标) | 🟢 先发者定义规则 |

### 2.5 壁垒生命周期

| 时间 | 壁垒状态 | 主要威胁 | 应对 |
|------|---------|---------|------|
| 0-6月 | 冯校长独占期 | 几乎没有 | 快速铺冯校长体系 |
| 6-12月 | 先发优势期 | 个别创业团队可能跟进 | 启动数据飞轮(10+店) |
| 12-18月 | 飞轮效应期 | 海康可能关注火锅 | 模型精度已拉开差距 |
| 18-24月 | 品牌壁垒期 | 大厂可能入局 | 冯校长标杆+行业案例 |
| 24月+ | 生态壁垒期 | 价格战 | 集成深度→高切换成本 |

## 三、失败模式分析

### 3.1 失败模式一：飞轮不启动（概率: 40%）

> **症状**: 3-6个月内无法达到10家付费门店，数据飞轮无法启动。

| 维度 | 分析 |
|------|------|
| **触发条件** | 冯校长试点效果不及预期、客户转化率<20%、获客成本失控 |
| **早期信号** | 试用→付费转化率<30%、客户留存<6个月、推荐率为0 |
| **根本原因** | AI减损效果不如预期、定价ROI不成立、客户付费意愿被高估 |
| **影响** | 从"产品公司"降级为"冯校长内部工具"——商业闭环断裂 |
| **概率** | **40%**（早期最高风险） |
| **严重度** | 🔴 致命 |
| **缓解措施** | A) 试点期设定严格里程碑(M1/3/6月Go/No-Go) B) 若效果不达标，快速转向纯SaaS轻模式 C) 冯校长内部消化(即使不对外，自用也省总代自己的钱) |
| **止损策略** | 如M6客户<3家+月亏损持续，停止对外推广，转为冯校长内部工具 |

### 3.2 失败模式二：政府免费AI挤压（概率: 25%）

> **症状**: 政府明厨亮灶项目覆盖火锅商户，"AI后厨应该免费"的认知固化，商业定价难以维系。

| 维度 | 分析 |
|------|------|
| **触发条件** | 政府项目扩展到社会餐饮(目前98.5%在学校食堂)、地方政策强制餐饮装AI |
| **早期信号** | 浙江省/台州市出台社会餐饮AI政策、美团/饿了么强制后厨直播 |
| **根本原因** | 政府有动力推食品安全监管，火锅商户若被纳入，火瞳的"运营价值"差异不足以支撑付费 |
| **概率** | **25%**（中期风险，目前政府项目在社会餐饮渗透<1%） |
| **严重度** | 🟡 严重（但非致命——可转型为政府项目集成商/差异化运营增值） |
| **缓解措施** | A) 强化"运营省钱"差异化信息——"政府管食安，火瞳管利润" B) 与政府项目互补而非对抗——火瞳API对接政府平台，成为"政府监管+商业运营"双模方案 C) 若政策强制，火瞳可申请成为政府指定服务商 |
| **止损策略** | 若政策强制覆盖，转型为"政府平台上的增值运营模块"，仅对连锁客户收费 |

### 3.3 失败模式三：一人公司带宽瓶颈（概率: 30%）

> **症状**: 客户增长超过个人服务能力，交付/运维/售后跟不上，口碑崩盘。

| 维度 | 分析 |
|------|------|
| **触发条件** | 同时维护>15家店的部署/升级/故障响应，或新客户涌入速度>个人交付速度 |
| **早期信号** | 客户投诉响应>24h、部署延期>2周、线上问题>1天未修复 |
| **根本原因** | 单人无法同时做产品开发+客户交付+售后运维+销售 |
| **概率** | **30%**（中期风险，规模扩大后必然出现） |
| **严重度** | 🟡 严重（但可控——提前设计自动化+阶段性扩张） |
| **缓解措施** | A) 10家店后引入AI Agent自动化运维（健康检查/告警/日报自动生成） B) 15家店后外包安装部署 C) 20家店后招第一个全职 D) 设备标准化(预装Jetson→快递到店→远程配置) E) 建立FAQ/知识库降低重复咨询 |
| **止损策略** | 设定硬上限：在未扩张团队前暂缓新客户签约(如月增≤3家)，优先保证存量客户服务质量 |
| **扩张路线** | 1人(MVP)→1人+AI Agent(10-20家)→2人(20-50家)→3人+AI(50-100家) |

### 3.4 失败模式概率矩阵

| 失败模式 | 概率 | 严重度 | 风险等级 | 发生时间 | 可逆性 |
|---------|:---:|:---:|:---:|:---:|:---:|
| 飞轮不启动 | 40% | 🔴 致命 | **高** | 0-6月 | 可逆(转内部工具) |
| 政府AI挤压 | 25% | 🟡 严重 | **中** | 12-24月 | 可逆(转型) |
| 带宽瓶颈 | 30% | 🟡 严重 | **中** | 6-18月 | 可逆(招人/AI) |
| 竞品切入 | 15% | 🟡 严重 | **低** | 12-18月 | 难逆 |
| 技术瓶颈 | 10% | 🟢 中等 | **低** | 3-12月 | 可逆(降scope) |

### 4.2 时机窗口量化

## 五、结论

### 核心判断

| 判断 | 结论 |
|------|------|
| **技术可行吗？** | ✅ 可行。YOLO+VLM基础链路已验证，火锅专有差距可工程化解（无基础研究级难题） |
| **商业可行吗？** | ✅ 可行。5店盈亏平衡合理，总代模式获客效率最优 |
| **时机对吗？** | ✅ 极好。商业后厨AI真空，火锅垂直无人做，12-18个月独占窗口 |
| **最大风险？** | ⚠️ 飞轮不启动（40%概率）——最大风险是"客户不来"而非"做不出来" |
| **能做多大？** | 3年1,000店（SOM），年收入¥1,500万。长期可跨品类→全餐饮 |
| **应该做吗？** | ✅ **Go**。条件：M6里程碑严格审查 |

### 一句话

> **火瞳在技术上没有硬障碍，最大的风险是客户获取速度能否跑赢窗口期。飞轮不启动 = 失败；飞轮启动 = 壁垒不可逆。**
